# Alert Notification System

This document describes the current notification subsystem as it exists in the codebase today.

## Current Role

The notification system is a delivery layer for alerts generated by the pipeline.

Current channels supported in code:

- email
- Slack

Notifications are optional and depend on environment configuration.

## Current Trigger Model

Notifications are driven by alert results from the alert evaluator.

Current alert rule types supported by the evaluator include:

- `PRICE_MOVEMENT`
- `MA_CROSSOVER`
- `RSI`
- `MACD`
- `VOLATILITY`
- `VOLUME_SPIKE`

Current delivery behavior:

- CRITICAL alerts may be sent individually
- WARNING alerts can be included in digest-style summaries

## Configuration

Current environment variables used by the settings layer:

```bash
NOTIFICATION_EMAIL_ENABLED=true
SMTP_HOST=smtp.gmail.com
SMTP_PORT=587
SMTP_USER=your_email@gmail.com
SMTP_PASSWORD=your_app_password
NOTIFICATION_FROM_EMAIL=alerts@stockpipeline.com
NOTIFICATION_EMAILS=recipient1@example.com,recipient2@example.com

NOTIFICATION_SLACK_ENABLED=true
SLACK_WEBHOOK_URL=https://hooks.slack.com/services/YOUR/WEBHOOK/URL
```

## Current Operational Notes

- notifications are best-effort and should not be allowed to break the pipeline
- notification success does not validate alert correctness
- alerts still depend on the quality of the underlying market data and indicator pipeline

## How It Is Used Today

The normal usage path is:

1. pipeline evaluates alerts
2. alert records are saved
3. notifier sends enabled channel outputs

## Current Caveat

This subsystem is current, but the surrounding alert and recommendation architecture is still under refinement. Treat notifications as monitoring aids, not trading instructions.
